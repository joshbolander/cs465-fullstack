# cs465-fullstack
CS-465 Full Stack Development with MEAN

**Architecture
Compare and contrast the types of frontend development you used in your full stack project, including Express HTML, JavaScript, and the single-page application (SPA). Why did the backend use a NoSQL MongoDB database?**
Throughout this project, I worked with two different styles of frontend development, which made it easier to see how each approach impacts both performance and user experience. The first approach used Express with HTML and JavaScript, where most of the rendering happens on the server. In that setup, every time a user navigates to a new page or performs an action, the server processes the request and sends back a fully rendered page. While this method is fairly straightforward and easier to follow when first learning full stack development, it can feel slower because the browser has to reload the entire page each time something changes.
The second approach used Angular to build a single-page application (SPA), which works very differently. Instead of reloading pages, Angular updates the interface dynamically by only changing the parts of the screen that need to be updated. This creates a much smoother and faster experience for the user, especially when switching between views like browsing trips, editing data, or adding new entries. It also allows the application to be structured more cleanly using components and services, which makes the code easier to manage and reuse. Overall, the SPA approach felt more modern and better suited for a responsive application compared to the traditional Express-based frontend.
The backend uses MongoDB as a NoSQL database because it aligns well with the type of data being handled in this application. Travel-related data, such as trips and itineraries, does not always fit neatly into a fixed table structure, so using a document-based database provides more flexibility. MongoDB stores data in JSON-like documents, which makes it easier to work with when sending data between the backend and the Angular frontend. This also fits naturally with the MEAN stack, allowing all parts of the application to communicate more efficiently without requiring complex data transformations.

**Functionality
How is JSON different from JavaScript and how does JSON tie together the frontend and backend development pieces? Provide instances in the full stack process when you refactored code to improve functionality and efficiencies, and name the benefits that come from reusable user interface (UI) components.**
One of the key concepts in this project is understanding how JSON differs from JavaScript and how the two work together. JavaScript is a full programming language that includes logic, functions, and behavior, while JSON is simply a format used to structure and transfer data. JSON does not contain logic, it is just a way of organizing information into key-value pairs. In this application, JSON acts as the communication layer between the frontend and backend. When the Angular application sends a request to the server, the server responds with JSON data, which the frontend then uses to update the user interface.
This separation allows the frontend and backend to remain loosely connected while still working together effectively. For example, when retrieving trip data, the backend sends a JSON response containing all the necessary details, and the Angular components use that data to display information on the screen. This makes it easier to update or modify one part of the system without breaking the other, since they communicate through a standardized format.
During development, I refactored parts of the code to improve both functionality and efficiency. One example was moving API calls into a centralized Angular service instead of placing them directly inside multiple components. This reduced code duplication and made it easier to maintain the application, since any changes to API endpoints only needed to be updated in one place. I also used reusable UI components for things like forms and trip displays. This helped keep the interface consistent and reduced the amount of repeated code. The main benefit of this approach is that it makes the application more modular, easier to debug, and more scalable as new features are added.

**Testing
Methods for request and retrieval necessitate various types of API testing of endpoints, in addition to the difficulties of testing with added layers of security. Explain your understanding of methods, endpoints, and security in a full stack application.**
Testing was an important part of making sure the application worked correctly across all layers of the system. Since the project relies heavily on API communication, I focused on testing the different HTTP methods used to interact with the backend. These methods include GET for retrieving data, POST for creating new records, PUT for updating existing data, and DELETE for removing records. Each method corresponds to a specific type of operation, and testing them helped confirm that the application was handling data correctly.
Endpoints act as the connection points between the frontend and backend. For example, an endpoint like /api/trips is used to retrieve all trip data, while /api/trips/:tripId is used for specific operations on a single trip. By testing these endpoints, I was able to verify that requests were being processed correctly and that the expected data was being returned or modified in the database.
Adding authentication introduced additional challenges to the testing process. Certain endpoints required valid login credentials before they could be accessed, which meant I had to test both authorized and unauthorized scenarios. For example, I needed to confirm that users without proper authentication could not access admin features, while authenticated users could successfully perform actions like editing or deleting trips. This helped reinforce how security is layered into a full stack application and how important it is to protect sensitive functionality. Overall, testing gave me a better understanding of how methods, endpoints, and security all work together to ensure a reliable and secure application.
